# Content Categorization App - Developer Guide

## Project Overview

- **Purpose**: Content categorization application for transcribing and categorizing video content using AI
- **Language**: Java 24
- **Framework**: Spring Boot 3.5.0
- **Database**: PostgreSQL with Flyway migrations
- **Architecture**: Layered architecture with clean separation of concerns
- **Authentication**: JWT-based authentication with Google OAuth2 integration

## Project Structure

The project follows a clean layered architecture:

### Core Packages

- **`domain`** - Core business logic and rules
  - `domain.model` - Pure domain models (User, Transcript, Category, etc.)
  - `domain.service` - Domain service interfaces
- **`data`** - Data persistence and external service interactions
  - `data.entity` - JPA entities with database annotations
  - `data.repository` - JPA repository implementations
  - `data.client` - External service clients (OpenAI, Whisper)
- **`application`** - Coordinates between domain and data layers
  - `application.mapper` - Maps between domain models, entities, and DTOs
  - `application.internal` - Internal application services
- **`api`** - HTTP communication layer
  - `api.controller` - REST controllers
  - `api.dto` - API Data Transfer Objects
- **`config`** - Configuration classes and beans
- **`security`** - Security configuration and JWT handling
- **`exception`** - Custom exceptions and global error handling
- **`util`** - Utility classes and helper methods

### Important Files & Directories

- **`pom.xml`** - Maven configuration with all dependencies
- **`src/main/resources/application.properties`** - Main configuration
- **`src/main/resources/application-secrets.properties`** - Secret configuration (gitignored)
- **`src/main/resources/db/migration/`** - Flyway database migrations
- **`src/test/resources/application-test.properties`** - Test configuration


### Running the Application

```bash
./mvnw spring-boot:run
```

- **Swagger UI**: http://localhost:8080/swagger-ui.html
- **OpenAPI JSON**: http://localhost:8080/v3/api-docs

## Coding Conventions & Best Practices

### Architecture Patterns

- **Clean Architecture**: Clear separation between domain, application, and infrastructure layers
- **Domain-Driven Design**: Pure domain models without framework dependencies
- **Repository Pattern**: Abstract data access through interfaces
- **Mapper Pattern**: Explicit mapping between layers using dedicated mapper classes

### Code Style

- **Lombok Usage**: Use `@Data`, `@Getter`, `@AllArgsConstructor` to reduce boilerplate
- **Constructor Injection**: Prefer constructor injection over field injection
- **UUID Primary Keys**: Use UUID for all entity primary keys
- **Validation**: Use Bean Validation annotations (`@Valid`, `@NotNull`, etc.)
- **Logging**: Use SLF4J with proper log levels and structured messages

### Entity vs Domain Model Separation

- **Domain Models**: Pure Java objects in `domain.model` package (no JPA annotations)
- **Entities**: JPA-annotated classes in `data.entity` package for persistence
- **Mapping**: Use dedicated mapper classes to convert between layers

### API Design

- **RESTful Endpoints**: Follow REST conventions for URL structure
- **Request/Response DTOs**: Separate DTOs for API communication
- **Error Handling**: Global exception handling with `@RestControllerAdvice`
- **Validation**: Comprehensive request validation with meaningful error messages
- **Documentation**: Use OpenAPI annotations for API documentation

### Security Implementation

- **JWT Authentication**: Token-based authentication with refresh tokens
- **Google OAuth2**: Integration for user authentication
- **User Context**: Proper user context handling throughout the application
- **Security Configuration**: Centralized security configuration in `SecurityConfig`

### Database Practices

- **Flyway Migrations**: All schema changes through versioned migration files
- **UUID Primary Keys**: Consistent use of UUIDs across all tables
- **Proper Indexing**: Strategic database indexes for performance
- **Constraint Naming**: Clear and consistent constraint naming

### Testing Practices

- **Testcontainers**: Use for integration tests with real PostgreSQL
- **Layer Testing**: Test each layer independently
- **Security Testing**: Include security-specific test scenarios
- **Test Configuration**: Separate test configuration with appropriate settings
- Group tests together in a file by nesting the tests of the method that is being tested. Do not nest further just one layer.

## Common Development Tasks

### Adding New Endpoints
1. Create domain model in `domain.model`
2. Create entity in `data.entity` 
3. Create repository interface in `domain.service` and implementation in `data.repository`
4. Create mapper in `application.mapper`
5. Create controller in `api.controller`
6. Add appropriate DTOs in `api.dto`

### Database Changes
1. Create new migration file in `src/main/resources/db/migration/`
2. Follow naming convention: `V{version}__{description}.sql`
3. Update corresponding entities
4. Update domain models if needed
5. Update mappers

### Adding External Service Integration
1. Create client interface in `domain.service`
2. Create implementation in `data.client`
3. Add configuration in `config` package
4. Add mock implementation for development mode
