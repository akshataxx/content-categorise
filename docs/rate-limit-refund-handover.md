# Rate Limit Refund on Pipeline Failure — Handover

> 🤖 This document was generated by Rovodev, the author has reviewed it for accuracy 🤖

## Context

As part of fixing Red Team Finding #3 (TOCTOU race condition), we moved `rateLimitService.recordTranscription(userId)` to **submission time** — before the job is created or the pipeline runs. This closes the race window but means a failed transcription still consumes a rate limit slot, penalising users for server-side failures.

## Goal

Refund the rate limit slot when the transcription pipeline fails due to **non-user-caused errors** (yt-dlp failures, Whisper errors, network issues, etc.). Do NOT refund for validation failures — those are caught before `recordTranscription()` is called.

## Proposed Changes

### 1. Add `refundTranscription(UUID userId)` to `RateLimitService`

- **Interface**: `domain/service/RateLimitService.java` — add method signature
- **Implementation**: `application/internal/RateLimitServiceImpl.java` — decrement the tracking counter
- This is the inverse of `recordTranscription()` — look at how that method increments `UserRateLimitTrackingEntity` and do the reverse
- Should be idempotent / safe if count is already 0 (don't go negative)

### 2. Call refund in `VideoController.handleVideo()` error path

**File**: `api/controller/VideoController.java`

In the `whenComplete` callback, the error branch currently does:
```java
} else {
    Throwable cause = (ex instanceof CompletionException) ? ex.getCause() : ex;
    if (isNewJob) {
        transcriptionJobService.handleFailure(job,
                (cause instanceof Exception) ? (Exception) cause : new RuntimeException(cause));
    }
}
```

Add `rateLimitService.refundTranscription(userId);` in this branch.

### 3. Call refund in `JobPollerService.executeJob()` catch block

**File**: `domain/service/JobPollerService.java`

In the `catch (Exception ex)` block, add the refund. But **do NOT refund** when the failure was the rate limit re-check itself (that path returns early before processing, and no extra slot was consumed by the poller).

Current code structure:
```java
private void executeJob(TranscriptionJobEntity job) {
    try {
        // Rate limit re-check — returns early on failure (no refund needed here)
        RateLimitResult rateLimitResult = rateLimitService.checkRateLimit(job.getUserId());
        if (!rateLimitResult.isAllowed()) {
            jobService.handleFailure(job, ...);
            return;  // <-- no refund, slot was consumed at submission time intentionally
        }

        // Pipeline runs...
        TranscriptDtoWithAliases result = videoService.processVideoAndCreateTranscript(...).join();
        ...
    } catch (Exception ex) {
        // <-- ADD refund here: pipeline failed after rate limit passed
        rateLimitService.refundTranscription(job.getUserId());
        jobService.handleFailure(job, ex);
    }
}
```

### 4. Edge case: async job where rate limit re-check fails

When the poller re-checks rate limits and denies the job, the slot was already consumed at submission time. We have two options:
- **Option A**: Leave it consumed (simpler, slightly penalises user)
- **Option B**: Refund it since the work wasn't done (fairer, add refund before the `return` in the re-check block)

Recommend **Option B** for consistency — the user paid the slot but got nothing.
